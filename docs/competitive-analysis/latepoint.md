## Architecture

LatePoint is a monolithic WordPress plugin entrypoint backed by a wide `lib/` tree of controllers, helpers, models, mailers, misc domain objects, and view templates, plus separate `blocks/` and `public/` assets directories for frontend/admin presentation (`research/sources/carte/latepoint/latepoint.php:41`, `research/sources/carte/latepoint/latepoint.php:697`, `research/sources/carte/latepoint/latepoint.php:858`).

The request lifecycle is classic WP-plugin orchestration: activation creates tables and schedules housekeeping, `init` registers post types/shortcodes/blocks, and nearly every runtime interaction funnels through a single router action exposed to both authenticated and unauthenticated AJAX plus admin-post endpoints (`research/sources/carte/latepoint/latepoint.php:889`, `research/sources/carte/latepoint/latepoint.php:898`, `research/sources/carte/latepoint/latepoint.php:923`, `research/sources/carte/latepoint/latepoint.php:1161`).

> `research/sources/carte/latepoint/latepoint.php:923-926`
```php
add_action( 'wp_ajax_latepoint_route_call', array( $this, 'route_call' ) );
add_action( 'wp_ajax_nopriv_latepoint_route_call', array( $this, 'route_call' ) );
```

The booking funnel itself is a routed step engine: public actions like `start`, `load_step`, and `load_datepicker_month` hydrate a mutable booking/cart state and render step partials, which makes the wizard extensible but also tightly couples UI flow to server-side session state (`research/sources/carte/latepoint/lib/controllers/steps_controller.php:14`, `research/sources/carte/latepoint/lib/controllers/steps_controller.php:36`, `research/sources/carte/latepoint/lib/controllers/steps_controller.php:177`, `research/sources/carte/latepoint/lib/views/steps/start.php:31`).

LatePoint’s hook surface is extensive and pragmatic. The core boot path exposes extension seams such as `latepoint_includes`, `latepoint_init_hooks`, and `latepoint_init`, while booking, order, payment, timeline, and settings views expose many `do_action`/`apply_filters` touchpoints for add-ons to override behavior without changing core files (`research/sources/carte/latepoint/latepoint.php:858`, `research/sources/carte/latepoint/latepoint.php:990`, `research/sources/carte/latepoint/lib/controllers/orders_controller.php:334`, `research/sources/carte/latepoint/lib/views/dashboard/widget_bookings_and_availability_timeline.php:117`, `research/sources/carte/latepoint/lib/views/services/_form.php:80`).

## Reservation domain model

The booking aggregate stores the expected reservation primitives directly on the booking row: service, customer, agent, status, start/end date-time fragments, duration, and before/after buffers. End times are derived from `start_time + total_duration`, and service defaults are copied onto bookings if not explicitly present (`research/sources/carte/latepoint/lib/models/booking_model.php:12`, `research/sources/carte/latepoint/lib/models/booking_model.php:152`, `research/sources/carte/latepoint/lib/models/booking_model.php:398`, `research/sources/carte/latepoint/lib/models/booking_model.php:423`, `research/sources/carte/latepoint/lib/models/booking_model.php:1024`).

Service rows carry the resource-planning rules that govern reservations: base duration, display/charge amounts, buffers, min/max capacity, optional custom durations, and per-agent/location connectivity. Capacity semantics are explicit enough to support both single-party appointments and group bookings (`research/sources/carte/latepoint/lib/models/service_model.php:13`, `research/sources/carte/latepoint/lib/models/service_model.php:23`, `research/sources/carte/latepoint/lib/models/service_model.php:135`, `research/sources/carte/latepoint/lib/models/service_model.php:168`, `research/sources/carte/latepoint/lib/models/service_model.php:400`).

> `research/sources/carte/latepoint/lib/models/service_model.php:135-139`
```php
public function get_capacity_needed_before_slot_is_blocked(): int{
  $capacity_min = $this->capacity_min ? $this->capacity_min : 1;
  $capacity_max = $this->capacity_max ? $this->capacity_max : 1;
  return ($this->get_meta_by_key('block_timeslot_when_minimum_capacity_met', 'off') == 'on') ? $capacity_min : $capacity_max;
}
```

The runtime booking request object is separate from the persisted booking model. It normalizes start/end, duration, buffers, attendees, and selected resource ids into a lightweight structure used for availability checks and slot generation, which is a strong separation between “reservation intent” and “saved reservation” (`research/sources/carte/latepoint/lib/misc/booking_request.php:10`, `research/sources/carte/latepoint/lib/misc/booking_request.php:28`, `research/sources/carte/latepoint/lib/misc/booking_request.php:44`).

Availability is not represented as a single table of slots. Instead, LatePoint composes daily resources from work periods, booked periods, blocked periods, and connector rules between services, agents, and locations. Shared-resource constraints such as “one agent at one location at a time” and “multiple services at time” are enforced while building daily resources, before slots are generated (`research/sources/carte/latepoint/lib/helpers/resource_helper.php:70`, `research/sources/carte/latepoint/lib/helpers/resource_helper.php:124`, `research/sources/carte/latepoint/lib/helpers/resource_helper.php:137`, `research/sources/carte/latepoint/lib/helpers/resource_helper.php:152`, `research/sources/carte/latepoint/lib/helpers/resource_helper.php:167`).

The reservation settings UI makes durations, buffers, timeblock overrides, booking-status overrides, and agent/location assignment first-class admin concepts. That is useful evidence that restaurant operators expect these knobs even when the final UX should be simpler in Carte (`research/sources/carte/latepoint/lib/views/services/_form.php:61`, `research/sources/carte/latepoint/lib/views/services/_form.php:106`, `research/sources/carte/latepoint/lib/views/services/_form.php:112`, `research/sources/carte/latepoint/lib/views/services/_form.php:127`).

## Time-slot generation algorithm

LatePoint computes bookable slots at read time from resources; it does not appear to persist a static “slot table.” The algorithm loads work periods and booked periods for a date range, materializes daily resources, asks each resource to build bookable slots at a timeblock interval, then flattens/squashes overlapping resource slots into a sorted list for rendering (`research/sources/carte/latepoint/lib/helpers/resource_helper.php:70`, `research/sources/carte/latepoint/lib/helpers/resource_helper.php:167`, `research/sources/carte/latepoint/lib/helpers/resource_helper.php:182`, `research/sources/carte/latepoint/lib/helpers/booking_helper.php:479`).

At the day level, the calendar helper walks the generated `BookingSlot` objects, keeps only slots whose `available_capacity()` can fit the requested attendees, and emits per-minute capacity data for group services. The slot object itself tracks `min_capacity`, `max_capacity`, `min_capacity_to_be_blocked`, and `booked_capacity`, so a slot can stay open even when partially used (`research/sources/carte/latepoint/lib/helpers/calendar_helper.php:303`, `research/sources/carte/latepoint/lib/helpers/calendar_helper.php:314`, `research/sources/carte/latepoint/lib/misc/booking_slot.php:9`, `research/sources/carte/latepoint/lib/misc/booking_slot.php:39`, `research/sources/carte/latepoint/lib/misc/booking_slot.php:47`).

Conflict detection is overlap-based rather than decrement-counter-based. `is_timeframe_in_booked_periods()` compares the buffered requested period against overlapping booked periods, sums existing attendees for the same service, and only blocks when the combined attendee count crosses the service’s configured threshold (`research/sources/carte/latepoint/lib/helpers/booking_helper.php:1152`, `research/sources/carte/latepoint/lib/helpers/booking_helper.php:1167`, `research/sources/carte/latepoint/lib/helpers/booking_helper.php:1175`).

> `research/sources/carte/latepoint/lib/helpers/booking_helper.php:1167-1171`
```php
if ( ( $count_existing_attendees + $booking_request->total_attendees ) <= $service->get_capacity_needed_before_slot_is_blocked() ) {
  return false;
} else {
  return true;
}
```

Compared to PRD §`Time-slot generation`, LatePoint aligns with Carte on the most important principle: slot availability is computed at read time from current resource state rather than maintained as a stale pre-generated ledger (`PRD.md:648`, `research/sources/carte/latepoint/lib/helpers/calendar_helper.php:243`, `research/sources/carte/latepoint/lib/helpers/resource_helper.php:167`). The divergence is in the underlying signal: Carte’s PRD proposes a simple per-slot capacity lookup keyed by `(date, slot)` for restaurant reservations, while LatePoint derives availability from richer overlapping work/booked periods plus service/agent/location intersections and group-capacity math (`PRD.md:651`, `PRD.md:657`, `research/sources/carte/latepoint/lib/helpers/resource_helper.php:137`, `research/sources/carte/latepoint/lib/helpers/booking_helper.php:1158`).

**Conclusion: ADAPT.** Carte should keep the PRD’s read-time computation pattern, because this teardown confirms LatePoint also avoids persisted slot rows. But we should adapt two LatePoint ideas: (1) represent reservation intent separately from the stored reservation, and (2) preserve explicit capacity semantics for partially filled slots. We should reject LatePoint’s WordPress-era complexity around multi-resource slot squashing for v0.1 single-location dining, because Carte’s PRD can satisfy the launch scope with a simpler read-time capacity model (`research/sources/carte/latepoint/lib/misc/booking_request.php:28`, `research/sources/carte/latepoint/lib/misc/booking_slot.php:39`, `PRD.md:644`, `PRD.md:657`).

## Email/notification pipeline

LatePoint treats notifications as a configurable subsystem rather than hardcoded transactional templates. Notification types are registered by channel (`email`, `sms`, `whatsapp`), each type resolves a selected processor, and the generic send path dispatches through helper-specific implementations while preserving a final filter hook for addons (`research/sources/carte/latepoint/lib/helpers/notifications_helper.php:8`, `research/sources/carte/latepoint/lib/helpers/notifications_helper.php:27`, `research/sources/carte/latepoint/lib/helpers/notifications_helper.php:69`, `research/sources/carte/latepoint/lib/helpers/notifications_helper.php:106`).

Email itself is thin and WP-native. `OsMailer` centralizes HTML headers and sender configuration, then hands off to `wp_mail`, while customer-specific flows such as password-reset email rendering live in dedicated mailer subclasses (`research/sources/carte/latepoint/lib/mailers/mailer.php:8`, `research/sources/carte/latepoint/lib/mailers/mailer.php:16`, `research/sources/carte/latepoint/lib/mailers/mailer.php:27`, `research/sources/carte/latepoint/lib/mailers/customer_mailer.php:17`, `research/sources/carte/latepoint/lib/mailers/customer_mailer.php:35`).

Templates are discoverable and editable from the admin. `NotificationsController::templates_index()` groups templates by recipient type, and `OsNotificationsHelper::load_templates_for_action_type()` assembles defaults for events like booking created, booking updated, and booking start using shared email layouts and template variables (`research/sources/carte/latepoint/lib/controllers/notifications_controller.php:20`, `research/sources/carte/latepoint/lib/helpers/notifications_helper.php:117`, `research/sources/carte/latepoint/lib/helpers/notifications_helper.php:130`, `research/sources/carte/latepoint/lib/views/notifications/templates_index.php:24`, `research/sources/carte/latepoint/lib/views/shared/_template_variables.php:30`).

The settings area exposes notifications as a dedicated admin page, and even the “From” name/address UI is injected through a processor settings hook. That suggests the product is designed for extensible channels rather than a single fixed confirmation email path (`research/sources/carte/latepoint/lib/controllers/settings_controller.php:121`, `research/sources/carte/latepoint/latepoint.php:947`, `research/sources/carte/latepoint/latepoint.php:1001`).

## Admin UX patterns

LatePoint’s admin UX centers on calendars and operational dashboards rather than raw tables. The calendar page supports day/week/month/list views, overlaying service availability, and multi-select filtering by services, locations, and agents, which is a mature control surface for staff coordinators (`research/sources/carte/latepoint/lib/controllers/calendars_controller.php:54`, `research/sources/carte/latepoint/lib/controllers/calendars_controller.php:126`, `research/sources/carte/latepoint/lib/views/calendars/view.php:20`, `research/sources/carte/latepoint/lib/views/calendars/view.php:54`).

The day view combines a horizontally navigable month strip with a vertical per-agent schedule grid. Each cell can launch a quick order/booking, and existing bookings are rendered as colored blocks over working-hour periods, which is a strong model for compact staff scheduling visibility (`research/sources/carte/latepoint/lib/views/calendars/scopes/_day.php:12`, `research/sources/carte/latepoint/lib/views/calendars/scopes/_day.php:30`, `research/sources/carte/latepoint/lib/views/calendars/scopes/_day.php:82`, `research/sources/carte/latepoint/lib/views/calendars/scopes/_day.php:121`).

The dashboard reuses the same operational primitives in lighter-weight widgets. The index action composes an availability timeline, daily-bookings chart, and upcoming appointments list, which gives managers a “day preview” home surface before they drill into the full calendar (`research/sources/carte/latepoint/lib/controllers/dashboard_controller.php:25`, `research/sources/carte/latepoint/lib/controllers/dashboard_controller.php:36`, `research/sources/carte/latepoint/lib/controllers/dashboard_controller.php:51`, `research/sources/carte/latepoint/lib/views/dashboard/widget_bookings_and_availability_timeline.php:5`).

The “Day Preview” widget is especially instructive: it lets operators toggle between appointments and availability, filter by location/service, and scan an agent-by-agent timeline with current-time indication and inline booking blocks (`research/sources/carte/latepoint/lib/views/dashboard/widget_bookings_and_availability_timeline.php:5`, `research/sources/carte/latepoint/lib/views/dashboard/widget_bookings_and_availability_timeline.php:17`, `research/sources/carte/latepoint/lib/views/dashboard/widget_bookings_and_availability_timeline.php:31`, `research/sources/carte/latepoint/lib/views/dashboard/widget_bookings_and_availability_timeline.php:76`).

The same calendar stack also powers customer self-service rescheduling and cabinet flows, which means LatePoint gets good reuse from one slot-rendering subsystem but also ties public UX closely to the same admin-era primitives (`research/sources/carte/latepoint/lib/views/shared/request_reschedule_calendar.php:23`, `research/sources/carte/latepoint/lib/controllers/customer_cabinet_controller.php:172`, `research/sources/carte/latepoint/lib/controllers/manage_booking_by_key_controller.php:152`).

## Strengths

1. **LatePoint models reservation intent separately from the persisted booking row.** The `BookingRequest` object gives the availability engine a clean boundary for calculating slots without mutating the stored booking until the user commits (`research/sources/carte/latepoint/lib/misc/booking_request.php:28`).
2. **Capacity semantics are explicit enough for group reservations.** `BookingSlot` and `ServiceModel` track both maximum capacity and the threshold at which a slot becomes blocked, enabling partially filled slots instead of binary occupancy (`research/sources/carte/latepoint/lib/misc/booking_slot.php:39`, `research/sources/carte/latepoint/lib/models/service_model.php:135`).
3. **The hook surface is broad and well-placed.** Core boot, step rendering, booking lifecycle, and timeline rendering all expose extension points, which lowers the need for forks in a plugin ecosystem (`research/sources/carte/latepoint/latepoint.php:990`, `research/sources/carte/latepoint/lib/controllers/orders_controller.php:334`, `research/sources/carte/latepoint/lib/views/dashboard/widget_bookings_and_availability_timeline.php:117`).
4. **Operational UX is built around staff workflows, not just CRUD tables.** The day/week/month calendar and dashboard timeline prioritize what operators need during service hours: who is working, what is booked, and where gaps remain (`research/sources/carte/latepoint/lib/controllers/calendars_controller.php:126`, `research/sources/carte/latepoint/lib/views/calendars/scopes/_day.php:82`, `research/sources/carte/latepoint/lib/views/dashboard/widget_bookings_and_availability_timeline.php:5`).
5. **Notifications are channel-aware and template-driven.** A single notification abstraction can dispatch email, SMS, or WhatsApp using configurable processors, while admins can preview event templates before enabling them (`research/sources/carte/latepoint/lib/helpers/notifications_helper.php:27`, `research/sources/carte/latepoint/lib/helpers/notifications_helper.php:69`, `research/sources/carte/latepoint/lib/views/notifications/templates_index.php:51`).

## Weaknesses

1. **The architecture is highly monolithic and WordPress-coupled.** A single plugin class wires activation, routing, UI asset loading, cron, auth, and payment hooks, which makes change impact broad and hard to reason about (`research/sources/carte/latepoint/latepoint.php:41`, `research/sources/carte/latepoint/latepoint.php:865`).
2. **The routing surface is too generic.** Multiplexing most product behavior through one `latepoint_route_call` endpoint obscures explicit contract boundaries and would be hard to secure or cache in a modern edge-first architecture (`research/sources/carte/latepoint/latepoint.php:923`, `research/sources/carte/latepoint/latepoint.php:1161`).
3. **Availability computation is powerful but operationally dense.** Slot generation depends on work periods, blocked periods, booked periods, resource intersections, status filters, and multiple boolean scheduling settings, which is likely over-complex for Carte’s v0.1 single-location scope (`research/sources/carte/latepoint/lib/helpers/resource_helper.php:70`, `research/sources/carte/latepoint/lib/helpers/resource_helper.php:137`, `research/sources/carte/latepoint/lib/helpers/booking_helper.php:1158`).
4. **Email delivery still bottoms out in WordPress mail primitives.** The notification system is configurable, but its default email path remains `wp_mail`, which limits observability and reliability compared with a first-class transactional mail boundary (`research/sources/carte/latepoint/lib/mailers/mailer.php:16`, `research/sources/carte/latepoint/lib/helpers/notifications_helper.php:84`).
5. **Admin flexibility comes with a lot of operator-facing configuration.** Calendars, services, notifications, work periods, capacities, and booking settings expose many low-level switches, which is powerful for consultants but risks overwhelming restaurant operators who want opinionated defaults (`research/sources/carte/latepoint/lib/views/calendars/view.php:54`, `research/sources/carte/latepoint/lib/views/services/_form.php:61`, `research/sources/carte/latepoint/lib/controllers/settings_controller.php:121`).
