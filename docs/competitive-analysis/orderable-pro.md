## Cart and modifier UX

Orderable Pro treats modifiers as flat field groups attached to products, not as recursive trees. The addon editor registers a dedicated `orderable_addons` post type, stores one array of `_orderable_addon_fields` plus a sibling `_orderable_addon_conditions` blob per group, and then resolves all matching groups before rendering product options (`research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-field-groups.php:50`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-field-groups.php:124`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-field-groups.php:131`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:52`).

At render time the frontend loops each applicable group and each field in that group, but the supported field types are still single-level primitives: `select`, `visual_radio`, `visual_checkbox`, and `text`. There is no nested group traversal, dependency tree, or recursive pricing pass; every option hangs directly off one field and every chosen value is flattened into `orderable_fields[group][field]` request data (`research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:92`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:100`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:156`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:167`).

> `research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:92-96`
```php
switch ( $field['type'] ) {
	case 'select':
	case 'visual_checkbox':
	case 'visual_radio':
	case 'text':
```

The authoring UI reinforces that ceiling. Each option row only captures a label, one price, one default-selection flag, and an optional visual treatment (`none`, `image`, `color`); the “selected” control can deselect sibling radios, but there is no representation for nested sub-groups or conditional children inside a choice (`research/sources/carte/orderable-pro/inc/modules/addons-pro/templates/admin/fields-options.php:11`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/templates/admin/fields-options.php:18`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/templates/admin/fields-options.php:47`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/templates/admin/fields-options.php:70`).

Modifier values are also flattened on the way into the cart and order. Selected values become cart item meta, cart display strings, and order line-item metadata with an optional fee suffix, which keeps the UX simple for buyers and staff but makes nested semantics impossible without redesigning the payload shape (`research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:156`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:203`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:245`).

On mobile, the product-options affordance is explicit: the add-to-cart button is relabeled to `Select` and forced through a `product-options` trigger whenever addon groups exist, and helper buttons annotate products with `data-orderable-trigger="product-options"` plus product/variation ids for a dedicated option surface (`research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:130`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:145`, `research/sources/carte/orderable-pro/inc/class-helpers.php:61`).

> `research/sources/carte/orderable-pro/inc/class-helpers.php:61-65`
```php
data-orderable-trigger="product-options"
data-orderable-product-id="<?php echo esc_attr( $product->is_type( 'variation' ) ? $product->get_parent_id() : $product->get_id() ); ?>"
data-orderable-product-type="<?php echo esc_attr( $product->get_type() ); ?>"
data-orderable-variation-id="<?php echo esc_attr( $product->is_type( 'variation' ) ? $product->get_id() : 0 ); ?>"
```

This is strong evidence for a single-tier modifier model in Carte v0.1: Orderable gets a polished selection flow without introducing nested choice trees, and its data model would need material redesign to support true parent/child modifiers.

### Recommendation for OQ#11 (modifier complexity ceiling, v0.1)

- Adopt: single-tier modifier groups with optional min/max/default semantics, matching Orderable’s flat field-group pattern where one product resolves applicable addon groups and each field owns a direct option list (`research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:52`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/templates/admin/fields-options.php:47`).
- Reject: nested modifier groups in v0.1. Orderable’s option editor, cart payload, and order metadata all flatten selections to one field/value layer, which keeps mobile UX workable but shows how much new machinery nested groups would require (`research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:156`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:245`).
- Therefore Carte v0.1 ships single-tier modifier groups (matching PRD §`Data Model` `ModifierGroup`) and defers nested modifiers to v0.2.

## Checkout flow

Orderable Pro’s checkout strategy is “replace WooCommerce chrome, keep WooCommerce order/payment plumbing.” When the override setting is enabled, the module swaps the block checkout back to the shortcode path, replaces the page template with `woocommerce/orderable/checkout.php`, rewires standard checkout hooks, and relocates coupon, login, and payment sections into an Orderable-specific layout (`research/sources/carte/orderable-pro/inc/modules/checkout-pro/class-checkout-pro-override-checkout.php:17`, `research/sources/carte/orderable-pro/inc/modules/checkout-pro/class-checkout-pro-override-checkout.php:44`, `research/sources/carte/orderable-pro/inc/modules/checkout-pro/class-checkout-pro-override-checkout.php:47`, `research/sources/carte/orderable-pro/inc/modules/checkout-pro/class-checkout-pro-override-checkout.php:68`, `research/sources/carte/orderable-pro/inc/modules/checkout-pro/class-checkout-pro-override-checkout.php:95`, `research/sources/carte/orderable-pro/inc/modules/checkout-pro/class-checkout-pro-override-checkout.php:292`).

The template itself is mobile-first in structure. It leads with a compact “show/hide order summary” toggle carrying the live total, places the order summary before customer fields in the markup, and breaks the form into `contact`, `billing`, `shipping`, and `payment` sections that can collapse or restyle independently (`research/sources/carte/orderable-pro/woocommerce/checkout/form-checkout.php:37`, `research/sources/carte/orderable-pro/woocommerce/checkout/form-checkout.php:41`, `research/sources/carte/orderable-pro/woocommerce/checkout/form-checkout.php:47`, `research/sources/carte/orderable-pro/woocommerce/checkout/form-checkout.php:61`, `research/sources/carte/orderable-pro/woocommerce/orderable/checkout/contact-fields.php:10`).

> `research/sources/carte/orderable-pro/woocommerce/checkout/form-checkout.php:37-44`

<div class="orderable-checkout-mobile-header">
	<a href="javascript:void(0)" class="orderable-checkout-summary-toggle">
		<span class="orderable-checkout-summary-toggle_link--show"><?php esc_html_e( 'Show order summary', 'orderable-pro' ); ?></span>

Field composition is opinionated. Contact fields are peeled out of billing, placeholders replace most labels, billing/shipping company fields are removed, and the checkout button receives a full-width branded class via HTML replacement (`research/sources/carte/orderable-pro/inc/modules/checkout-pro/class-checkout-pro-override-checkout.php:182`, `research/sources/carte/orderable-pro/inc/modules/checkout-pro/class-checkout-pro-override-checkout.php:205`, `research/sources/carte/orderable-pro/inc/modules/checkout-pro/class-checkout-pro-override-checkout.php:246`, `research/sources/carte/orderable-pro/inc/modules/checkout-pro/class-checkout-pro-override-checkout.php:272`).

The order summary keeps WooCommerce’s pricing engine intact while surfacing restaurant-specific concerns. Shipping rows hide when the selected service is pickup, line items include formatted cart metadata (including addons), and fees/tax/coupon totals remain first-class in the summary table (`research/sources/carte/orderable-pro/woocommerce/checkout/review-order.php:82`, `research/sources/carte/orderable-pro/woocommerce/checkout/review-order.php:111`, `research/sources/carte/orderable-pro/woocommerce/checkout/review-order.php:118`, `research/sources/carte/orderable-pro/woocommerce/checkout/review-order.php:125`).

For blocks checkout, Orderable extends the WooCommerce Store API instead of bypassing it. It registers extra cart/checkout schema under `orderable-pro/order-service-time`, then saves service-time fields back onto the order during `woocommerce_store_api_checkout_update_order_from_request` (`research/sources/carte/orderable-pro/orderable-pro.php:230`, `research/sources/carte/orderable-pro/orderable-pro.php:255`, `research/sources/carte/orderable-pro/inc/modules/checkout-pro/blocks/order-time/class-checkout-pro-order-time-extend-store-api.php:17`, `research/sources/carte/orderable-pro/inc/modules/checkout-pro/blocks/order-time/class-checkout-pro-order-time-extend-store-api.php:53`).

This maps well to Carte’s Stripe Checkout plan: keep one canonical payments backend, but own the pre-payment restaurant-specific UI and metadata capture layer instead of forcing customers through a generic commerce checkout.

## Service hours / time-slot logic for delivery vs pickup

Orderable distinguishes day-level capacity from slot-level capacity. `Timings_Pro_Settings` adds `Max Orders (Day)` at the open-hours level, and per-slot settings capture hours, frequency, lead time (`cutoff`), and `Max Orders (Slot)` (`research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro-settings.php:49`, `research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro-settings.php:69`, `research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro-settings.php:101`, `research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro-settings.php:113`, `research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro-settings.php:125`).

Slot generation happens per service type. `get_slots()` loads service-hour rows for `delivery` or `pickup`, expands each rule into a `DatePeriod`, subtracts the larger of global lead time or slot cutoff, skips past slots, and then filters each candidate through `orderable_time_slot_available` before exposing it (`research/sources/carte/orderable-pro/inc/modules/multi-location-pro/class-location-single-pro.php:242`, `research/sources/carte/orderable-pro/inc/modules/multi-location-pro/class-location-single-pro.php:319`, `research/sources/carte/orderable-pro/inc/modules/multi-location-pro/class-location-single-pro.php:327`, `research/sources/carte/orderable-pro/inc/modules/multi-location-pro/class-location-single-pro.php:332`, `research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro.php:57`).

> `research/sources/carte/orderable-pro/inc/modules/multi-location-pro/class-location-single-pro.php:319-324`
```php
$cutoff_timestamp = $timestamp;
if ( ! empty( $lead_time ) || ! empty( $setting_row['cutoff'] ) ) {
	$cutoff_seconds    = (int) $setting_row['cutoff'] * 60;
	$cutoff_timestamp -= $lead_time > $cutoff_seconds ? $lead_time : $cutoff_seconds;
}
```

Availability checks are backed by actual order counts. `get_orders_remaining_for_time_slot()` excludes cancelled/refunded/failed orders, counts orders with matching `_orderable_order_timestamp` plus `_orderable_service_type`, and subtracts that from the configured slot max; date-level availability does the same for the full day window (`research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro.php:153`, `research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro.php:185`, `research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro.php:222`, `research/sources/carte/orderable-pro/inc/modules/multi-location-pro/class-location-single-pro.php:138`, `research/sources/carte/orderable-pro/inc/modules/multi-location-pro/class-location-single-pro.php:199`).

Checkout enforces the same rules twice: classic checkout validates posted `orderable_order_date` / `orderable_order_time`, while blocks checkout re-reads Store API payloads. Same-day slots must still satisfy the current time plus slot cutoff, and `asap` resolves to the first available slot on the chosen service date (`research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro-checkout.php:21`, `research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro-checkout.php:73`, `research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro-checkout.php:112`, `research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro-checkout.php:191`, `research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro-checkout.php:225`).

Delivery and pickup are also split at the location-selection layer. Postcode search returns separate arrays of locations that currently support delivery versus pickup, ETAs are calculated per service, and a location can pause delivery and pickup independently through per-service transients that expire at tomorrow’s boundary (`research/sources/carte/orderable-pro/inc/modules/multi-location-pro/class-multi-location-pro-search.php:30`, `research/sources/carte/orderable-pro/inc/modules/multi-location-pro/class-multi-location-pro-search.php:34`, `research/sources/carte/orderable-pro/inc/modules/multi-location-pro/templates/postcode-search-result-single-store.php:18`, `research/sources/carte/orderable-pro/inc/modules/multi-location-pro/templates/postcode-search-result-single-store.php:36`, `research/sources/carte/orderable-pro/inc/modules/multi-location-pro/class-location-single-pro.php:469`, `research/sources/carte/orderable-pro/inc/modules/multi-location-pro/class-location-single-pro.php:497`).

That split is directly relevant to Carte: service type is not cosmetic; it changes address collection, shipping fees, location eligibility, and capacity accounting.

## Order state machine and admin views

Orderable Pro layers a configurable state machine on top of WooCommerce orders. Custom statuses are stored as an `orderable_status` post type, registered as `wc-<slug>` statuses at runtime, and merged into WooCommerce’s status registry and reports (`research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro.php:16`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro.php:156`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro.php:189`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro.php:292`).

The admin editor exposes more than naming. Each status stores enablement, slug, icon, color, `nextstep` transitions, inclusion in reports, and a notification graph; bulk actions and row actions in the WooCommerce order table are then rewritten so staff can advance an order along only the configured next steps (`research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro-admin.php:144`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro-admin.php:164`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro-admin.php:287`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro-admin.php:310`).

Notifications are tightly coupled to status transitions. The metabox lets staff configure email, SMS, or WhatsApp notifications per status, per recipient, and optionally per location; the runtime `send_notifications()` hook fans out on `woocommerce_order_status_changed`, resolves recipients, and triggers either the custom WooCommerce email class or channel-specific actions (`research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/templates/admin/notifications-metabox.php:55`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/templates/admin/notifications-metabox.php:190`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro.php:341`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro.php:377`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro.php:412`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro-email.php:127`).

Orderable also suppresses WooCommerce’s default customer emails when a custom status already sends its own customer email, preventing duplicate notifications on transitions like `processing`, `on-hold`, and `completed` (`research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro.php:539`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro.php:561`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro.php:652`).

For operators, the admin view is richer than a raw status dropdown: colored badges, iconized action buttons, notification icons in the status list, and location-aware notification targeting all turn the order table into a lightweight operations board (`research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro-admin.php:243`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro-admin.php:269`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro-admin.php:364`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro-admin.php:411`).

## Strengths

1. **The checkout override is unapologetically mobile-first.** A summary toggle with live total, reordered sections, and an owned template give Orderable a better phone checkout than stock WooCommerce (`research/sources/carte/orderable-pro/woocommerce/checkout/form-checkout.php:37`, `research/sources/carte/orderable-pro/woocommerce/checkout/form-checkout.php:47`).
2. **Modifiers stay understandable because the data model is flat.** One field group resolves to one layer of product options, which keeps cart metadata and operator review legible (`research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:52`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:245`).
3. **Service-time logic respects real operational constraints.** Slot generation accounts for frequency, cutoff, service type, and per-slot/day max orders instead of just showing every clock time (`research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro-settings.php:113`, `research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro.php:141`, `research/sources/carte/orderable-pro/inc/modules/multi-location-pro/class-location-single-pro.php:319`).
4. **Location and service selection are first-class checkout concepts.** Delivery versus pickup changes eligible locations, ETAs, validation, and order metadata instead of being a late shipping-method afterthought (`research/sources/carte/orderable-pro/inc/modules/multi-location-pro/class-multi-location-pro.php:276`, `research/sources/carte/orderable-pro/inc/modules/multi-location-pro/templates/postcode-search-result-single-store.php:78`).
5. **The order-state admin is configurable enough for restaurant ops.** Staff can define custom statuses, next steps, reporting behavior, and notification bundles without hacking WooCommerce core flows (`research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro-admin.php:144`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro-admin.php:287`).

## Weaknesses

1. **The plugin entrypoint contains an obvious secret-handling violation.** The vendored main file writes a dashboard license key directly into `orderable_settings` on load, which is exactly the kind of mutable secret behavior Carte must avoid (`research/sources/carte/orderable-pro/orderable-pro.php:14`).
2. **Modifier flexibility hits a ceiling quickly.** Because selections flatten into one `orderable_fields` map and one label/value meta stream, nested modifier groups would require a structural rewrite rather than an incremental extension (`research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:156`, `research/sources/carte/orderable-pro/inc/modules/addons-pro/class-addons-pro-fields.php:245`).
3. **Checkout ownership comes with compatibility debt.** The override module ships a long list of theme and plugin compatibility shims, a sign that replacing WooCommerce checkout markup is powerful but expensive to maintain (`research/sources/carte/orderable-pro/inc/modules/checkout-pro/class-checkout-pro-override-checkout.php:20`, `research/sources/carte/orderable-pro/inc/modules/checkout-pro/flux-common/class-compat-avada.php:45`).
4. **Time-slot capacity is order-count based, not quantity or prep-load based.** `Max Orders (Slot)` subtracts counted orders, so a tiny coffee and a huge catering-style pickup consume the same slot budget (`research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro.php:141`, `research/sources/carte/orderable-pro/inc/modules/timings-pro/class-timings-pro.php:222`).
5. **Order-state configuration is powerful but WP-heavy.** A custom-post-type editor, Vue metaboxes, and per-status notification JSON are workable for consultants, but they are a lot of moving parts for restaurant operators who mostly need opinionated defaults (`research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/class-custom-order-status-pro-admin.php:16`, `research/sources/carte/orderable-pro/inc/modules/custom-order-status-pro/templates/admin/notifications-metabox.php:9`).
