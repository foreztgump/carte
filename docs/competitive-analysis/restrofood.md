## Domain coordination

RestroFood behaves like a bundle of subsystems wired together by one bootstrap path. The main plugin file seeds five WordPress pages (`restrofood`, `branch-manager`, `kitchen-manager`, `delivery`, `admin`) and a shared `restrofood_options` record on activation, then conditionally loads the actual feature set only when the license gate and WooCommerce dependency pass (`research/sources/carte/restrofood/restrofood.php:117`, `research/sources/carte/restrofood/restrofood.php:197`, `research/sources/carte/restrofood/restrofood.php:214`, `research/sources/carte/restrofood/restrofood.php:78`, `research/sources/carte/restrofood/restrofood.php:113`).

That bootstrap fans into a broad include graph instead of formal modules. `restrofood-init.php` pulls in delivery distance logic, order JSON builders, frontend-admin templates, Woo hooks, AJAX handlers, widgets, and the settings admin in one pass, so menu browsing, checkout metadata, branch operations, and frontend rendering all share one runtime namespace (`research/sources/carte/restrofood/restrofood-init.php:10`, `research/sources/carte/restrofood/restrofood-init.php:17`, `research/sources/carte/restrofood/restrofood-init.php:25`, `research/sources/carte/restrofood/restrofood-init.php:31`, `research/sources/carte/restrofood/restrofood-init.php:36`).

Orders are the main state-sharing hub. Checkout writes delivery type, pickup branch, pickup time, and delivery date directly onto WooCommerce order meta, and later filters reuse those same meta keys to drive branch dashboards, delivery dashboards, pre-order views, and operator notifications (`research/sources/carte/restrofood/inc/class-woo-hooks.php:137`, `research/sources/carte/restrofood/inc/class-woo-hooks.php:145`, `research/sources/carte/restrofood/inc/class-woo-hooks.php:193`, `research/sources/carte/restrofood/inc/class-woo-hooks.php:205`, `research/sources/carte/restrofood/frontend-admin/template-admin-components.php:549`).

Delivery scheduling is reservation-adjacent rather than a separate reservations domain. `Date_Time_Map::timeListMaping()` derives bookable delivery/pickup times from either global settings or branch-specific day schedules, checks holidays and breaks, and can block a slot once the order count reaches the configured per-slot limit (`research/sources/carte/restrofood/inc/class-date-time-map.php:51`, `research/sources/carte/restrofood/inc/class-date-time-map.php:64`, `research/sources/carte/restrofood/inc/class-date-time-map.php:187`, `research/sources/carte/restrofood/inc/helper-functions.php:1095`).

Branch and location context are shared through transients more than explicit contracts. Availability checks set `d_availability_status`, `rb_location_distance`, `rb_branch_id`, and `rb_selected_zipcode`, and later checkout fee calculation, branch selection, slot generation, and address display all reread those transients to decide pricing and eligibility (`research/sources/carte/restrofood/inc/class-components-ajax.php:1318`, `research/sources/carte/restrofood/inc/class-components-ajax.php:1321`, `research/sources/carte/restrofood/inc/helper-functions.php:1140`, `research/sources/carte/restrofood/inc/helper-functions.php:1215`, `research/sources/carte/restrofood/view/modal-checkout.php:24`).

The public product browser, branch-aware ordering, and operator interfaces all sit on the same AJAX mesh. Shortcodes and templates output lightweight placeholders, then `class-products.php` and `class-components-ajax.php` expose dozens of `wp_ajax_*` endpoints for product queries, cart mutations, date filtering, order status changes, branch transfer, time list generation, location checks, and delivery-boy assignment (`research/sources/carte/restrofood/inc/class-hooks.php:18`, `research/sources/carte/restrofood/inc/class-products.php:18`, `research/sources/carte/restrofood/inc/class-products.php:30`, `research/sources/carte/restrofood/inc/class-components-ajax.php:36`, `research/sources/carte/restrofood/inc/class-components-ajax.php:66`).

For Carte, the clearest lesson is architectural: RestroFood proves that restaurants want menu, order, and scheduling data to feel unified, but it accomplishes that by smearing state across one giant options blob, Woo order meta, and transients. Carte should keep the unified operator experience while maintaining explicit package boundaries and typed contracts between `@carte/core`, `@carte/orders-backend`, and `@carte/reservations`.

## Admin information architecture

RestroFood’s wp-admin IA is shallow at the top level but dense inside the settings screen. `Admin_Menu::admin_menu_page()` creates one root “Restrofood” menu with a default Settings entry, a dynamic hook for add-ons, and a single “Orders” submenu, so the sidebar footprint is intentionally small even though the plugin spans many operational concerns (`research/sources/carte/restrofood/admin/inc/class-admin-menu.php:33`, `research/sources/carte/restrofood/admin/inc/class-admin-menu.php:41`, `research/sources/carte/restrofood/admin/inc/class-admin-menu.php:42`).

Inside that settings page, the real information architecture is tab-driven. `Admin_Templates_Map::tab()` renders 12 tabs: General, Delivery Settings, Kitchen Options, Order Received Page, Page Settings, Color Settings, Location Settings, Email Settings, Status Text, Text Edit, Invoice Settings, and Shortcode List (`research/sources/carte/restrofood/admin/inc/admin-template.php:51`, `research/sources/carte/restrofood/admin/inc/admin-template.php:55`, `research/sources/carte/restrofood/admin/inc/admin-template.php:62`, `research/sources/carte/restrofood/admin/inc/admin-template.php:71`, `research/sources/carte/restrofood/admin/inc/admin-template.php:75`).

Those tabs mix structural routing with operational policy. Page Settings maps five public/internal pages (shop, branch manager, kitchen manager, delivery, admin), effectively making page wiring part of plugin configuration rather than code (`research/sources/carte/restrofood/admin/inc/tabs-content/page-settings.php:24`, `research/sources/carte/restrofood/admin/inc/tabs-content/page-settings.php:31`, `research/sources/carte/restrofood/admin/inc/tabs-content/page-settings.php:45`, `research/sources/carte/restrofood/admin/inc/tabs-content/page-settings.php:59`).

Delivery Settings is especially overloaded. It combines customer-facing service type choices, minimum order thresholds, free-shipping logic, pre-order toggles, day/time schedules, slot granularity, slot order caps, time zone, map routing mode, and closure text inside one tab (`research/sources/carte/restrofood/admin/inc/tabs-content/delivertimebranch.php:24`, `research/sources/carte/restrofood/admin/inc/tabs-content/delivertimebranch.php:46`, `research/sources/carte/restrofood/admin/inc/tabs-content/delivertimebranch.php:69`, `research/sources/carte/restrofood/admin/inc/tabs-content/delivertimebranch.php:96`, `research/sources/carte/restrofood/admin/inc/tabs-content/delivertimebranch.php:142`, `research/sources/carte/restrofood/admin/inc/tabs-content/delivertimebranch.php:154`).

Location Settings is similarly broad: it controls whether delivery availability checking exists at all, whether the plugin uses address or zip-code matching, which Google API key to use, how far delivery can reach, which pages should show the modal, and whether checkout must re-run the location checker (`research/sources/carte/restrofood/admin/inc/tabs-content/location.php:24`, `research/sources/carte/restrofood/admin/inc/tabs-content/location.php:31`, `research/sources/carte/restrofood/admin/inc/tabs-content/location.php:38`, `research/sources/carte/restrofood/admin/inc/tabs-content/location.php:60`, `research/sources/carte/restrofood/admin/inc/tabs-content/location.php:89`, `research/sources/carte/restrofood/admin/inc/tabs-content/location.php:101`).

Beyond wp-admin, RestroFood also ships its own frontend-admin surfaces for branch managers, kitchen managers, and delivery staff. The template router swaps dedicated pages to `template-branch-management.php`, `template-kitchen-management.php`, `template-delivery-management.php`, and a general admin template, while `template-admin-components.php` renders statistics cards, order tables, and modal workflows for those roles (`research/sources/carte/restrofood/inc/class-hooks.php:144`, `research/sources/carte/restrofood/inc/class-hooks.php:152`, `research/sources/carte/restrofood/inc/class-hooks.php:160`, `research/sources/carte/restrofood/frontend-admin/template-admin-components.php:111`, `research/sources/carte/restrofood/frontend-admin/template-admin-components.php:403`).

This directly informs Carte’s four planned admin pages in `@carte/core`: RestroFood shows that keeping the sidebar compact is good, but it also shows the cost of overloading one settings hub. Carte should split structural admin destinations from policy-heavy settings instead of burying every concern under one monolithic tab strip.

## Frontend rendering approach

RestroFood’s storefront rendering is shortcode and template first, with JavaScript filling in the dynamic data. `Hooks::restrofood_shortcode()` and `Hooks::restrofood_flash_sale_shortcode()` collect shortcode attributes, stash them in globals, and include PHP views that output wrapper markup plus `data-options` JSON for later hydration (`research/sources/carte/restrofood/inc/class-hooks.php:69`, `research/sources/carte/restrofood/inc/class-hooks.php:83`, `research/sources/carte/restrofood/inc/class-hooks.php:101`, `research/sources/carte/restrofood/view/template-part-woo-items.php:19`, `research/sources/carte/restrofood/view/template-part-woo-items.php:69`).

The page router reinforces that pattern. Instead of blocks or components mounted from a manifest, `template_include` swaps full-page templates for the shop, frontend-admin, branch manager, kitchen manager, and delivery pages (`research/sources/carte/restrofood/inc/class-hooks.php:128`, `research/sources/carte/restrofood/inc/class-hooks.php:139`, `research/sources/carte/restrofood/inc/class-hooks.php:152`, `research/sources/carte/restrofood/inc/class-hooks.php:168`).

Modal UX is a first-class rendering primitive. `rb_js_template()` injects Underscore.js modal templates for the wrapper, product content, login/register, reviews, billing summary, alerts, and order modal into the footer, then separately includes `modal-cart.php`, which decides between canvas and footer-fixed cart shells (`research/sources/carte/restrofood/inc/class-hooks.php:248`, `research/sources/carte/restrofood/inc/class-hooks.php:254`, `research/sources/carte/restrofood/inc/class-hooks.php:257`, `research/sources/carte/restrofood/view/template-modal-wrapper.php:1`, `research/sources/carte/restrofood/view/modal-cart.php:14`).

Checkout is rendered as a custom overlay around WooCommerce, not as a native Woo page. `modal-checkout.php` builds a delivery/pickup step, date selector, time selector, optional availability checker, and then drops Woo billing/shipping/order-review hooks into the same card layout (`research/sources/carte/restrofood/view/modal-checkout.php:13`, `research/sources/carte/restrofood/view/modal-checkout.php:24`, `research/sources/carte/restrofood/view/modal-checkout.php:77`, `research/sources/carte/restrofood/view/modal-checkout.php:153`, `research/sources/carte/restrofood/view/modal-checkout.php:184`).

The product listing view is deliberately shell-like. `template-part-woo-items.php` renders search/filter controls, branch selectors, sort options, and either a standalone products container or a side-by-side cart layout; the actual product cards and cart state are expected to be filled by AJAX responses and JS behavior (`research/sources/carte/restrofood/view/template-part-woo-items.php:74`, `research/sources/carte/restrofood/view/template-part-woo-items.php:88`, `research/sources/carte/restrofood/view/template-part-woo-items.php:104`, `research/sources/carte/restrofood/view/template-part-woo-items.php:117`).

Widgets are an alternate entry point into that same rendering system. `Elementor_Init` registers two Elementor widgets, and `Products_Card` ultimately emits the same `[restrofood_products]` shortcode with layout and mini-cart parameters, so page-builder usage is just a wrapper around the shortcode/template pipeline (`research/sources/carte/restrofood/widgets/Elementor_Init.php:15`, `research/sources/carte/restrofood/widgets/Elementor_Init.php:30`, `research/sources/carte/restrofood/widgets/Products_Card.php:26`, `research/sources/carte/restrofood/widgets/Products_Card.php:208`).

Thank-you and account-adjacent views continue the override strategy. The plugin replaces WooCommerce’s thank-you page with a branded order-details, invitation, and sharing layout, extending the same “restaurant-first shell around WooCommerce core” philosophy to post-purchase surfaces (`research/sources/carte/restrofood/template/woocommerce/checkout/thankyou.php:20`, `research/sources/carte/restrofood/template/woocommerce/checkout/thankyou.php:56`, `research/sources/carte/restrofood/template/woocommerce/checkout/thankyou.php:160`, `research/sources/carte/restrofood/template/woocommerce/checkout/thankyou.php:211`).

For Carte, the adoptable idea is not the shortcode stack itself, but the separation between a stable shell and data-driven filling. In Astro + native admin, we should preserve the explicit composition of product browser, cart, checkout, and post-order surfaces while replacing global PHP templates and footer script injection with typed components and route contracts.

## Strengths

1. **RestroFood understands that restaurant operators want one coherent workflow across menu browsing, ordering, and service-time selection.** The activation path creates the public shop plus branch, kitchen, delivery, and admin pages together, signaling a unified operating model rather than disconnected plugins (`research/sources/carte/restrofood/restrofood.php:148`).
2. **Order meta is used consistently as the cross-domain contract.** Delivery type, branch, date, and time are written once at checkout and then reused by dashboards, filtering, and thank-you/order-detail views (`research/sources/carte/restrofood/inc/class-woo-hooks.php:137`, `research/sources/carte/restrofood/frontend-admin/template-admin-components.php:549`).
3. **Slot generation is read-time and branch-aware.** Delivery/pickup times are generated from current schedules, breaks, holidays, and slot caps instead of being hardcoded or statically stored (`research/sources/carte/restrofood/inc/class-date-time-map.php:83`, `research/sources/carte/restrofood/inc/class-date-time-map.php:187`).
4. **The admin surface keeps sidebar sprawl low.** One root menu plus an Orders submenu is a good reminder that complex restaurant products do not need a dozen top-level WordPress menu items (`research/sources/carte/restrofood/admin/inc/class-admin-menu.php:33`, `research/sources/carte/restrofood/admin/inc/class-admin-menu.php:42`).
5. **Frontend rendering is deliberately composable.** Shortcodes, page-template swaps, modal shells, and Elementor widgets all reuse the same product/cart/checkout primitives, which reduces duplication across entry points (`research/sources/carte/restrofood/inc/class-hooks.php:69`, `research/sources/carte/restrofood/inc/class-hooks.php:248`, `research/sources/carte/restrofood/widgets/Products_Card.php:208`).

## Weaknesses

1. **The plugin is monolithic even by WordPress standards.** One include pass wires almost every domain into one runtime, making menu, location, checkout, dashboards, widgets, and notifications hard to reason about independently (`research/sources/carte/restrofood/restrofood-init.php:10`, `research/sources/carte/restrofood/restrofood-init.php:36`).
2. **State is fragmented across options, order meta, and transients without clear boundaries.** Delivery availability, chosen branch, selected zip code, and fee context live in transient keys that unrelated code later assumes exist, which is brittle under concurrency and hard to test (`research/sources/carte/restrofood/inc/class-components-ajax.php:1318`, `research/sources/carte/restrofood/inc/helper-functions.php:1140`, `research/sources/carte/restrofood/inc/helper-functions.php:1215`).
3. **The settings IA is overloaded.** A 12-tab settings page plus frontend-admin pages centralizes too many structural, operational, and cosmetic controls, which likely increases setup burden for small restaurants (`research/sources/carte/restrofood/admin/inc/admin-template.php:55`, `research/sources/carte/restrofood/admin/inc/tabs-content/delivertimebranch.php:24`, `research/sources/carte/restrofood/admin/inc/tabs-content/location.php:24`).
4. **The frontend stack relies on global PHP state and footer-injected templates.** Globals like `$restrofoodAttr`, runtime `data-options` blobs, and injected Underscore templates make the rendering pipeline flexible but difficult to type, cache, or isolate (`research/sources/carte/restrofood/inc/class-hooks.php:79`, `research/sources/carte/restrofood/view/template-part-woo-items.php:64`, `research/sources/carte/restrofood/inc/class-hooks.php:248`).
5. **The license/update path is a security and product-boundary smell.** The main plugin file writes a license key into options on load, and the bundled base class short-circuits license checks to always return a fabricated “valid” response in the vendored source, which is the opposite of the server-verified, fail-open trial model Carte wants for `@carte/ai` (`research/sources/carte/restrofood/restrofood.php:20`, `research/sources/carte/restrofood/restrofoodBase.php:520`).
