# `@carte/core`

Status: **skeleton (v0.1)** — no business logic yet.

The Carte core plugin will own:

- Menu items, sections, and menus
- Restaurant settings (currency, locale, timezone, address)
- Hours of operation
- Schema.org JSON-LD output for menus and the restaurant entity

Execution model: **sandboxed** (Cloudflare Worker per invocation). Capabilities declared:
`content:read`, `content:write`, `media:read`. No outbound network, no email.

## Data retention and erasure

Carte core provides admin-only GDPR routes for guest data export and erasure.
Reservations and orders retain revenue records for accounting and analytics:
totals, currency, party size, and order line-item snapshots are preserved.

Guest PII in reservation `guest` records and order `customer` records is erased
by replacing `name`, `email`, `phone`, and `notes` with a deterministic
`erased:<sha256>` placeholder derived from the normalized email address. This
keeps same-guest analytics linkable during the retention period without keeping
the original contact details.

Retention periods are:

- Reservation and order financial records: 7 years, or the restaurant's longer
  statutory accounting period.
- Export files generated by admins: transient download only; do not store them
  in Carte.
- Operational logs containing route status: 30 days, with no guest contact
  details logged by these handlers.

On Cloudflare Free plans, EmDash cannot host sandboxed plugins with Dynamic
Workers, so plugins run trusted instead of isolated. The erasure behavior is the
same, but restaurants should treat the whole Worker as trusted application code.
