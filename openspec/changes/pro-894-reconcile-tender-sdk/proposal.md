# pro-894-reconcile-tender-sdk — Reconcile `@tender/sdk` dependency shape for publish-before-Tender

## Why

`@carte/orders-backend` must publish to npm **before** `@tender/sdk@^0.1.0`
exists on the registry (R1, gated only on itself). Today the package declares
`@tender/sdk` as a `devDependency` pointing at a relative vendored tarball
(`file:../../vendor/tender-sdk-0.0.0.tgz`). That `file:` shape is published
verbatim in the package manifest (npm/pnpm do not strip `devDependencies` from a
packed tarball), and `MIGRATION.md` separately documents the SDK as a _peer
dependency_ — a documented-vs-actual mismatch (gap **G-PAY-EXT**). Both are
release blockers: a clean install of the packed tarball must not depend on a
path or registry version that does not exist for the consumer.

## What Changes

- **Reconcile to a self-contained bundle.** `@tender/sdk` becomes a build-time-only
  input that `emdash-plugin build` inlines into `dist/plugin.mjs` /
  `dist/index.mjs` (already the build's behavior — the SDK is imported by
  `src/routes/checkout.ts` and `src/routes/refund.ts` and is not externalized).
  The **published** `@carte/orders-backend` declares **no** `@tender/sdk`
  dependency in any field (no `dependencies`, `devDependencies`,
  `peerDependencies`, or `optionalDependencies`).
- **Relocate the vendored build/test input** off the published package so build,
  typecheck, and test still resolve `@tender/sdk` from a workspace input that is
  never published (the workspace root is `private: true`).
- **Fix the documented shape.** Update `MIGRATION.md` (and the orders-backend
  README where it states the dependency contract) to describe the SDK as
  bundled/self-contained — operators install `tender-core` / `tender-stripe`
  (separate plugins) for payments to function, **not** `@tender/sdk` directly.
- **Add a pack-contents verification** that asserts the packed tarball carries no
  `@tender/sdk` reference of any kind, no `vendor/` path, and no `src/`, and that
  a clean install of the packed tarball with `@tender/sdk` absent exits 0.
- **No runtime/handler code changes.** Checkout/refund logic is unchanged; this
  is a packaging and dependency-shape reconciliation only.

This is **not** the optional-peer approach (considered and rejected in
`design.md`): the SDK is already inlined into `dist/`, so an optional peer would
be redundant and risk a dual-version mismatch.

## Capabilities

- **Modified:** `release-packaging` — adds requirements covering the
  `@carte/orders-backend` `@tender/sdk` dependency shape, clean-install behavior,
  and self-contained pack contents (delta in
  `specs/release-packaging/spec.md`).

## Impact

- `packages/orders-backend/package.json` — remove the `@tender/sdk`
  `devDependency`; `files` allowlist already excludes `src/` and the vendored
  tarball (no change needed there, asserted by the new spec).
- `package.json` (workspace root) — carry the vendored `@tender/sdk` build/test
  input (`file:./vendor/tender-sdk-0.0.0.tgz`); root is `private: true` → never
  published.
- `.npmrc` — make the hoisted SDK resolvable from the package under pnpm strict
  `node_modules` (existing `public-hoist-pattern[]` convention).
- `pnpm-lock.yaml` — regenerated by the install.
- `scripts/` — new pack-contents + clean-install verification script
  (mirrors the existing `scripts/verify-rich-results.ts` /
  `scripts/audit-sandbox-budget.ts` pattern).
- Docs (implementation-time, per `tasks.md`): `MIGRATION.md`,
  `packages/orders-backend/README.md`, `CHANGELOG.md` / changeset.

## Fulfills

- PRD-production-release **WS-A4** (load-bearing): installing
  `@carte/orders-backend` from npm does not error while `@tender/sdk@^0.1.0` is
  absent; the `devDependency` + `file:` tarball shape does not leak into the
  published tarball.

## Rollback plan

Revert the `package.json` (package + root), `.npmrc`, and `scripts/` edits and
restore the prior `pnpm-lock.yaml`. `dist/` is gitignored and the vendored
`.tgz` is unchanged, so deleting the worktree fully reverts. The package is
unpublished (`0.3.0-rc.1`, `npm view` → E404) with no external consumers, so the
dependency-shape change cannot break an existing install.
